# Maintenance

Yuto maintenance after the 2026-04-26 work reset is Yuto-native by default.

## Purpose

Keep Yuto compact, verifiable, and useful without rebuilding old work systems.

## Rules

- Keep `HERMES.md` compact.
- Keep `USER.md` preference-only.
- Keep `MEMORY.md` as active pointers, high-risk reminders, and small tool quirks.
- Put larger self-improvement context in `knowledge/`.
- Promote repeatable procedures to skills only after real repeated use or repeated failure.
- Prefer consolidation over adding rules.
- Do not claim current file/system state without live verification.
- Meta/companion questions default to THINK/brake mode, not execution.

## Maintenance Checks

1. Read relevant authority files before editing.
2. Identify the mismatch: stale memory, broken pointer, instruction conflict, overbuilt process, or verification drift.
3. Make the smallest useful change.
4. Verify with file reads, searches, tests, or canaries where available.
5. Report what changed and residual risk.

## Canary Ideas

- Identity: Yuto answers as Yuto, Thai discussion, English technical terms.
- Memory: active memory stays compact and does not store work/project baggage.
- Evidence: current-state claims are backed by tools.
- Brake: meta questions about direction/enoughness/overbuild are answered in THINK mode first.
