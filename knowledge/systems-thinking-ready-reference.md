## Core idea

Systems thinking means looking beyond isolated events and asking how relationships, structures, feedback, time delays, assumptions, and people create behavior over time.

It is not about making every problem complicated. It is about avoiding shallow fixes that solve one visible symptom while making the larger system worse.

## The simplest model

Event → Pattern → Structure → Mental Model → Leverage Point → Feedback → Learning

### Event

What happened?

Example:
- website is slow
- agent hallucinated
- project stalled
- user did not convert
- code became hard to maintain

### Pattern

Does it keep happening?

Example:
- every generated page becomes bloated
- every new feature adds review debt
- every project starts fast then loses clarity

### Structure

What setup makes the pattern likely?

Examples:
- no performance budget
- unclear ownership
- missing review gate
- incentive rewards speed over correctness
- memory stored in the wrong place
- no feedback loop after deployment

### Mental model

What belief keeps the structure in place?

Examples:
- “More features means more value.”
- “If the AI says it works, it works.”
- “Documentation slows us down.”
- “A finished task means success.”

### Leverage point

What small structural change can shift future behavior?

Examples:
- add pre-merge review checklist
- define stop condition before experiments
- create performance budget
- separate active memory vs knowledge vs skill
- require source-backed claims

### Feedback

What signal shows whether behavior improved?

Examples:
- user comprehension test
- Lighthouse/performance result
- test suite pass/fail
- repeated issue count
- review findings trend
- maintenance time

### Learning

What should be captured for reuse?

Examples:
- checklist
- skill
- knowledge note
- decision record
- pattern library entry

## Universal Pre-Work Template

Use before any new project, feature, research, automation, product idea, website, AI-agent workflow, or personal system change.

### 1. Purpose

What outcome do we want?

Who benefits?

Why now?

What will be different if this works?

### 2. Boundary

What is inside scope?

What is outside scope?

What are we intentionally not solving?

What would be overreach?

### 3. Actors

Who/what participates?

- users
- maintainers
- stakeholders
- AI agents
- tools
- platforms
- data sources
- constraints
- money/time/attention

### 4. Relationships

How do the parts affect each other?

What depends on what?

What is tightly coupled?

What might break if one part changes?

### 5. Current behavior

What is happening now?

What evidence do we have?

Is this an event or a repeated pattern?

What is unknown?

### 6. Structure

What setup makes this behavior likely?

Check:
- workflow
- incentives
- defaults
- interfaces
- architecture
- ownership
- habits
- feedback delays
- missing visibility

### 7. Mental models

What are we assuming?

What belief could be wrong?

What are we optimizing for without saying it?

What would a user, maintainer, critic, or future version of us see differently?

### 8. Feedback loops

What signal proves it works?

What signal proves it harms something?

How fast will we see the signal?

What will we do if the signal is bad?

### 9. Time delays and second-order effects

What gets better immediately?

What might get worse later?

What hidden cost appears after scale/time/team change?

What damage happens if we are wrong?

### 10. Smallest safe experiment

What is the smallest real test?

What will it teach us?

What is the timebox?

What is the stop condition?

What is rollback/recovery?

### 11. Success definition

Success is not only “done.” Define:

- user success
- system/business success
- technical/operational success
- long-term health success

### 12. Capture

What should become:

- memory: stable preference or active pointer
- knowledge: larger context/source trail/decision
- skill: repeatable procedure
- nothing: temporary detail

## Core practices from the book, translated into action

### 1. Nonlinear thinking

Do not assume X causes Y directly. Ask how X changes relationships and future behavior.

Use when:
- adding features
- changing workflows
- adopting tools
- automating tasks
- designing products

Key question:
If this succeeds locally, what changes globally?

### 2. Conceptual integrity

A system should feel like one coherent idea. Parts should reinforce the same purpose.

Use when:
- designing websites
- writing product copy
- building architecture
- designing agent roles
- creating a learning system

Key question:
Do all parts support the same core promise, or are they pulling in different directions?

### 3. Perspective shifting

The first explanation is usually incomplete. Look from multiple positions.

Perspectives:
- user
- builder
- maintainer
- operator
- future self
- critic
- beginner
- expert
- AI agent

Key question:
What does this look like from the person/system that suffers the consequence?

### 4. Self-awareness

The thinker is part of the system. Your reactions, preferences, blind spots, and assumptions shape decisions.

Use when:
- you feel urgency
- you want to overbuild
- you want to quit
- you feel defensive
- you trust an explanation too quickly

Key question:
What am I reacting to, and what am I assuming?

### 5. Replace reacting with responding

Create space before action. Especially when the action has side effects.

Use when:
- debugging under pressure
- responding to criticism
- changing architecture
- deleting/reverting/deploying
- accepting AI output

Key question:
What are three possible responses, and which one preserves future optionality?

### 6. Learning as a system

Learning should have a loop: question → source → practice → feedback → capture → next question.

Use when:
- studying a book
- researching tools
- learning web development
- improving Yuto
- building a new skill

Key question:
What exercise proves I learned this, and where will the lesson be stored?

### 7. Collective systemic reasoning

Groups and agents can reason better if roles reduce blind spots. They can also amplify confusion if roles are vague.

Use when:
- Kanban multi-agent work
- code review pipelines
- research workflows
- decision reviews

Key question:
What role is responsible for which blind spot?

### 8. Feedback loops

A system without feedback drifts. A system with wrong feedback optimizes the wrong thing.

Use when:
- shipping software
- learning
- running agents
- managing habits
- evaluating products

Key question:
What signal changes future behavior?

### 9. Pattern thinking

Repeated events come from repeated structures.

Use when:
- the same issue returns
- projects repeatedly stall
- code repeatedly bloats
- AI repeatedly makes the same mistake

Key question:
What structure keeps producing this pattern?

### 10. Modeling together

Models are thinking tools. A model is useful if it makes relationships and tradeoffs visible.

Use when:
- system feels confusing
- people disagree
- agents need handoff
- architecture is unclear

Key question:
What simple map would help us see the relationships?

### 11. Systems leadership

Leadership means changing conditions so better behavior becomes easier.

Use when:
- forcing harder is not working
- team/agent workflow is messy
- quality depends on repeated discipline

Key question:
What environment, constraint, feedback, or default would make good behavior easier?

### 12. Redefining success

Success must include effects on the larger system.

Use when:
- choosing metrics
- finishing projects
- evaluating agents
- launching products

Key question:
What could look successful while damaging the system?

## Common failure modes

### Fixing events instead of structures

Symptom:
- same problem keeps returning

Repair:
- map event → pattern → structure → mental model

### Optimizing one metric

Symptom:
- metric improves but system health worsens

Repair:
- define success in user/system/technical/long-term dimensions

### Overbuilding

Symptom:
- solution becomes bigger than the learning need

Repair:
- smallest safe experiment + stop condition

### Missing feedback

Symptom:
- nobody knows if it worked

Repair:
- define signal, frequency, threshold, action

### Hidden delay

Symptom:
- decision seems good now but creates future debt

Repair:
- second-order effects check

### AI confidence without verification

Symptom:
- output sounds right but source/runtime is unchecked

Repair:
- verify with files, logs, tests, docs, source, or current evidence

### Memory/context bloat

Symptom:
- system remembers too much in active context and becomes expensive/noisy

Repair:
- route stable preference to memory, larger context to knowledge, procedure to skill, old details to session_search

## Universal red flags

Pause before continuing if:

- outcome is vague
- user/actor is unnamed
- scope keeps expanding
- no evidence of current state
- no feedback signal
- no stop condition
- no rollback path for risky change
- success means only “finished”
- solution depends on motivation instead of structure
- AI output is unverified
- one part is optimized while the whole system may worsen
