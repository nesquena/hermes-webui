## Cross-Source Patterns

### 1. Brake before architecture

Source links:
- [[source-learning-systems-thinking]] frames value as shifting from reacting to responding and seeing events as part of deeper patterns.
- [[source-agentic-architectural-patterns]] treats agentic systems as maturity progressions requiring governance.

Yuto implication:
Before building agents or automations, classify whether Kei needs THINK, RESEARCH, PLAN, or EXECUTE. Meta/direction/feeling questions stay in THINK unless Kei explicitly asks for execution.

### 2. Agent roles should emerge from capability and coordination need

Source links:
- [[source-agentic-architectural-patterns]] provides coordination patterns: router, supervisor, blackboard, contract-net, swarm, guarded hierarchy.
- [[source-30-agents-every-ai-engineer-must-build]] maps agents by capability: retrieval, planning, memory, tool-using, document intelligence, research, orchestration.

Yuto implication:
If Yuto later splits into agents, the first question is not "what persona?" but:

1. What capability is missing?
2. What coordination pattern contains the risk?
3. What tools/data are allowed?
4. What evaluation proves the agent is useful?
5. What stop condition prevents runaway work?

### 3. KG first, RAG/vector later

Source links:
- [[source-learning-systems-thinking]] emphasizes pattern/structure over event accumulation.
- [[source-30-agents-every-ai-engineer-must-build]] notes retrieval systems need metadata, reranking, observability.
- [[source-agentic-architectural-patterns]] treats RAG/context as an adaptation option, not identity.

Yuto implication:
Markdown KG remains source of truth. Generated graph/RAG/vector layers are indexes. Do not move to semantic RAG until the current notes have enough reuse and evaluation questions.

### 4. Multimodal/frontdoor is separate from Yuto core

Source links:
- [[source-multimodal-real-time-ai-agent-systems]] separates live interaction architecture from core agent logic and highlights streaming, tool, and AgentOps boundaries.
- [[source-learning-systems-thinking]] supports intentional response design instead of raw reaction.

Yuto implication:
Future voice/video/screen features should feed labeled observations into Yuto, not become automatic memory. Raw multimodal input is untrusted and must not auto-promote into active memory.

### 5. AgentOps can stay lightweight until repeated use proves need

Source links:
- [[source-multimodal-real-time-ai-agent-systems]] frames AgentOps as evaluation, registries, memory/data governance, and operational controls.
- [[source-agentic-architectural-patterns]] highlights observability/security/compliance as production concerns.
- [[source-30-agents-every-ai-engineer-must-build]] frames agent engineering as lifecycle + governance.

Yuto implication:
Current lightweight equivalents are enough for now:

- [[yuto-growth-loop]] for promotion gates
- [[workflows]] for recurring behavior
- [[sources]] for source trails
- graph/canary checks for integrity
- Completion Contract for execution closure

Do not create a heavy always-on AgentOps platform yet.

## Yuto Architecture Heuristic

When Kei asks to grow Yuto, use this ordering:

1. Systems thinking: What pattern/structure is causing the need?
2. Source grounding: What book/doc/source supports the pattern?
3. Architecture pattern: Is this router, supervisor, blackboard, contract-net, swarm, or guarded hierarchy?
4. Capability role: retrieval, planning, memory, research, document intelligence, tool execution, evaluation, or frontdoor?
5. Safety gate: what must not happen automatically?
6. Verification: what metric/canary proves the loop helped?
