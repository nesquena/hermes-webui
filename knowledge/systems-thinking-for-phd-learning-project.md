## Core framing

A PhD-entry project is not just a project artifact.

It is a system of:

- research question
- prior literature
- personal learning loop
- technical prototype
- evidence and evaluation
- writing/explanation quality
- advisor/reviewer expectations
- time and energy
- portfolio signal
- future research direction

The goal is not only to finish something impressive. The goal is to build a project that demonstrates research taste, disciplined inquiry, technical ability, and ability to learn from feedback.

## The PhD project question

Before choosing or building anything, ask:

What system behavior am I trying to understand, improve, or explain?

A good PhD-oriented project usually has:

1. A real problem or tension
2. A clear system boundary
3. Prior work to compare against
4. A small but meaningful intervention
5. Evidence that the intervention changed something
6. A written explanation of limits and implications

## Weekly PhD learning loop

Use once per week.

### Monday — question

What question am I trying to answer this week?

Why this question now?

What would change if I answer it?

### Tuesday/Wednesday — source + model

What did I read?

What model/diagram did I build?

What changed in my understanding?

### Thursday — experiment

What small test/prototype did I run?

What evidence did I collect?

### Friday — synthesis

What did I learn?

What is still uncertain?

What should be written down?

What is next week's question?

## Red flags for PhD-project work

Pause if:

- project is impressive but question is vague
- prototype exists but evaluation is weak
- reading grows but no model/prototype appears
- claim is stronger than evidence
- scope expands every week
- success means only “I built it”
- no comparison to prior work
- no limitation section
- no feedback from another person/system
- learning is not captured in notes

Use this to begin a PhD project discussion:
