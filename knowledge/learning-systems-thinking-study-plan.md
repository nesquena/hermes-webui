## Core thesis for Kei

Systems thinking is a practice for reasoning about nonlinear behavior: relationships, feedback, time delays, incentives, mental models, and sociotechnical effects. For software professionals, it helps avoid fixing symptoms while worsening the system.

## 12-lesson track

### Lesson 0 — Orientation / why this book matters

Goal: set a shared language for nonlinear software systems.

Focus:
- systems vs components
- events vs patterns vs structure vs mental models
- why obvious fixes often backfire
- why this matters for Yuto, Hermes, AI swarm, product, architecture

Exercise:
- Pick one recurring problem in Kei's work/life system.
- Describe it as: event, pattern, structure, mental model.

### Lesson 1 — Chapter 1: What Is Systems Thinking?

Goal: shift from linear cause-effect thinking to nonlinear relationship thinking.

Practice:
- For any problem, ask: “What relationships produce this behavior?” not only “What caused this?”

### Lesson 2 — Chapter 2: Crafting Conceptual Integrity

Goal: understand why coherent systems depend on relationships, communication, and time.

Practice:
- Map one product or agent workflow as a system of ideas, roles, feedback, and delays.

### Lesson 3 — Chapter 3: Shifting Your Perspective

Goal: learn the iceberg model and perspective shifting.

Practice:
- Take one frustrating event and map: events → patterns/trends → structures → mental models.

### Lesson 4 — Chapter 4: Self-Awareness as a Foundational Skill

Goal: treat the thinker as part of the system.

Practice:
- Notice assumptions, blind spots, emotional signals, and preferred explanations.

### Lesson 5 — Chapter 5: Replace Reacting with Responding

Goal: create space between trigger and action.

Practice:
- For a triggering technical/product problem, write three possible responses before acting.

### Lesson 6 — Chapter 6: A System of Learning

Goal: design learning as a system, not random consumption.

Practice:
- Build a learning loop: question → source → experiment → reflection → memory/skill.

### Lesson 7 — Chapter 7: Collective Systemic Reasoning

Goal: use teams/agents as a collective reasoning system.

Practice:
- Design a multi-agent Kanban workflow where each role reduces a specific blind spot.

### Lesson 8 — Chapter 8: Designing Feedback Loops

Goal: identify reinforcing and balancing loops.

Practice:
- Draw feedback loops for Yuto memory, project momentum, or AI-agent quality control.

### Lesson 9 — Chapter 9: Pattern Thinking

Goal: recognize repeated structures behind repeated outcomes.

Practice:
- Build a pattern catalog: symptom, recurring structure, leverage point, anti-pattern.

### Lesson 10 — Chapter 10: Modeling, Together

Goal: use modeling as shared thinking, not just documentation.

Practice:
- Create a simple shared model for a real system: actors, flows, feedback, delays, constraints.

### Lesson 11 — Chapter 11: Systems Leadership

Goal: lead by improving conditions for better system behavior.

Practice:
- Identify where to change constraints, incentives, visibility, or feedback instead of pushing harder.

### Lesson 12 — Chapter 12: Redefining Success

Goal: choose success measures that do not damage the larger system.

Practice:
- For one project, list success metrics and possible second-order harms.

## Practical applications for Kei

### Yuto / Hermes / AI agents

- Avoid measuring only “task done”; measure verified outcome, residual risk, memory routing, and recurrence reduction.
- Use Kanban for long-running multi-agent work where feedback, blocking, audit, and handoff matter.
- Treat active memory as a constraint; route durable context to knowledge/skills.

### Software architecture

- Ask what system behavior the architecture encourages.
- Look for feedback loops: deploy speed, defect rate, review quality, incident learning.
- Model sociotechnical forces: ownership, incentives, cognitive load, hidden coupling.

### Product decisions

- Do not optimize one visible metric without checking downstream effects.
- Use system maps before major product/UX/automation decisions.

### Personal operating system

- Track patterns, not only events.
- Design feedback loops for energy, learning, focus, and recovery.
